# State schema — the source of truth

Every topic is one folder under `<topics_dir>/<slug>/` (from `config.yaml`,
default `yoda/<slug>/`):

```
yoda/kubernetes/
├── state.json     ← authoritative. The agent edits ONLY this.
├── map.html       ← generated. The navigable map the learner opens.
└── map.md         ← generated. Readable mirror for editor / GitHub diffs.
```

**The agent edits `state.json` and nothing else.** `map.html` and `map.md` are
always regenerated by `scripts/build_map.py` — never hand-write or hand-patch
them. This is what keeps the artifact from drifting or breaking: structured edits
to one small JSON file, deterministic rendering of the views.

Everything is idempotent: re-running `build_map.py` on unchanged state produces
identical output, and resuming a topic means reading `state.json`, not
reconstructing from chat.

## Field reference

**Top level** — `topic`, `slug`, `goal`, `level`, `one_sentence` (the whole
domain in a line; the map is suspect until this exists), `you_are_here` (id of the
region currently being learned), `created`/`updated` (ISO dates). Optional
`tier_labels` (4 strings) and `mastery_labels` (5 strings) override the ladder names
on the map — copy them from `config.yaml` if the user renamed them.

**terrain.regions[]** — `id` (stable slug), `title`, `summary` (one phrase),
`tier` (`foundation` | `core` | `advanced` | `frontier` — the map bands regions by
this, foundations at the base), `core` (optional bool for extra prominence), and
`models` (optional list of `{ name, note }` — the key theories/frameworks for the
region, shown in its drawer). Keep to 4–8 regions.

**terrain.edges[]** — `from`/`to` region ids and a short `label`; these draw the
map's connections. Optional but they're what make it a *map* and not a list.

**terrain.not_covered[]** — the edges of the territory. Naming what's *out* of
scope is half of what makes a learner feel oriented.

**chunks[]** — one idea. `status` is `locked` | `learning` | `learned`. `mastery`
is the learner's grip, **0–4** (0 Unseen · 1 Fragile · 2 Working · 3 Solid · 4
Expert) — set honestly after each answer; it's what the node badge shows. `gap` is
the one-line, blunt delta to the next level (empty at 4). `note` is the
one-paragraph essence. `recall_q` / `recall_a` power both the gate and future
reviews. `models` (optional) is the same `{ name, note }` list as regions, for
chunk-specific theories. `reviews` drives scheduling (below). `schema` (optional) is
a small box-and-arrow diagram shown in the node's detail drawer —
`{ "type": "flow" | "stack", "nodes": [...], "labels": [...] }` where `labels` (one
shorter than `nodes`) sit on the arrows; add it only for chunks whose shape is worth
seeing. **Planned-but-unlearned chunks are real entries** with `status: locked`,
`mastery: 0`, and no `reviews` — that's how the roadmap shows the whole curriculum
up front.

**grammars[]** — a compressed region. `variables` (2–4; if more, you haven't
compressed enough), `paragraph` (the grammar in prose), plus its own `reviews`.

**open_questions[]** — parked tangents. `resolved` flips true when answered;
resolved ones drop off the map's active panel but stay in state.

## Spaced repetition (SM-2-lite)

Each reviewable item (every chunk and grammar) carries a `reviews` object:

| field | meaning |
|-------|---------|
| `ease` | difficulty multiplier, default **2.3**, floor 1.3, cap 3.0 |
| `interval` | current gap in days |
| `reps` | consecutive successful reviews |
| `last_review` | ISO date of the last grade |
| `next_due` | ISO date the item is next due |
| `history` | list of `{date, grade}` |

**Do not compute these by hand** — the math is error-prone and you'll drift. Use
the `review` command (below); it applies this exact algorithm:

```
grade = hard:
    reps = 0
    interval = 1
    ease = max(1.3, ease - 0.20)

grade = ok:
    reps += 1
    interval = 1   if reps == 1
             = 4   if reps == 2
             = round(interval * ease)   otherwise
    # ease unchanged

grade = easy:
    reps += 1
    interval = 2   if reps == 1
             = 6   if reps == 2
             = round(interval * (ease + 0.15))   otherwise
    ease = min(3.0, ease + 0.10)

next_due = today + interval days
append {today, grade} to history
status -> "learned"
```

Repeated **ok** on a fresh item gives roughly 1 → 4 → 9 → 21 → 48 days — an
expanding curve that fights forgetting cheaply. **hard** always resets to
tomorrow and makes the item a little stickier (lower ease) so it comes back more
often until it's solid.

The **first** recall gate when a chunk is learned *is* its first grade: nailed it
unprompted → `easy`; correct with a nudge → `ok`; struggled → `hard`. So logging a
new chunk is: add it to `chunks[]` (status `learning`, no `reviews`), then run
`review` with the gate result — the script initializes the schedule and flips
status to `learned`.

## The build_map.py commands

```bash
# Regenerate map.html + map.md from state (run after every state edit)
python scripts/build_map.py build learn/kubernetes/state.json

# Apply a review grade to one item (chunk or grammar) — updates schedule, rebuilds
python scripts/build_map.py review yoda/kubernetes/state.json --item c1 --grade ok

# List items due on/before a date (default today) — for starting a session or a scheduled task
python scripts/build_map.py due learn/kubernetes/state.json
```

`review` and `due` both accept `--date YYYY-MM-DD` to override "today" (useful for
testing and for scheduled runs). All three are deterministic and safe to re-run.

## Editing rules for the agent

- Generate stable, unique ids (`c1`, `c2`, `g1`, `q1`). Never renumber existing
  ids — reviews and "you are here" reference them.
- Touch `updated` on every change.
- Append, don't rewrite: new chunks/questions are added; existing learned chunks
  are only modified through `review`.
- After any edit to `state.json`, run `build_map.py build`. The chat is never the
  record of truth — the file is.
